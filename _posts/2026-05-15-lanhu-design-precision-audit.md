蓝湖稿 UI 还原最常见的误区，是把验收简化为「让人眼确认与截图看起来差不多」。这种判断成本高、误差大：文本节点位置看似正确，父容器外边距可能依然错误；颜色十六进制值相同，Design Token 语义可能已经偏离；长列表截图看不出问题，短内容或空列表时才暴露底部空白底色错误。

更稳定的做法，是把 UI 验收拆解为机器可读的工作流。只有以蓝湖结构化数据为规格，以页面级盒模型、模块间距台账、背景归属链、颜色语义角色和结构元素存在性为检查对象，才能将大语言模型（LLM）或 AI Agent 引入验收环节，完成高精度的自动化 UI 审核。

---

## 痛点定位：截图阻碍了 AI 自动化

UI 还原的误差通常不源于单个属性的绝对值错误，而源于属性归属判断错误。

例如，设计稿显示首个内容模块距离顶部 Tab 为 `8dp`。代码中可能有以下几种实现方式：

- Tab 自身底部的 `padding`。
- 页面根布局的 `8dp` Spacer 占位。
- 首个内容模块的 `layout_marginTop=8dp`。
- 列表 Header 生成的间距。

再比如，设计稿要求页面使用某个「页面背景 Token」，代码中可能是 Activity 根、Fragment 根、Tab 容器、刷新容器、RecyclerView 父容器或 Item 根节点中的任意一层在提供最终可见颜色。长列表被 Item 填满时，截图可能看不出差异；内容不足一屏、空列表、加载态或切换 Tab 后，错误的父级背景才会露出来。

在单纯的截图对比中，这些实现方式的视觉结果可能完全相同。即使是先进的多模态视觉大模型（Vision Model），仅凭两张截图也很难判定底层实现是否合理。后续一旦涉及视图隐藏、列表复用、短内容、空状态或主题切换，布局和背景差异就会暴露。因此，让 AI 准确审核代码，不能仅输入视觉截图，必须提供精确的节点结构上下文。

---

## 原则：结构化数据是 AI 的最佳 Prompt

蓝湖 API 返回的 `HTML`、`CSS`、`tokens`、`layout`、`layers` 等结构化数据，是驱动 AI 进行审核的唯一可靠规格。

典型的 AI 解析顺序为：

```text
agent -> analyze(include: html, tokens, layout, layers, codebase)
```

对于 `detailDetach` 类型的 URL，AI 需要完整提取所有设计节点。将这些结构化信息作为 Prompt 注入，AI 就能明确以下工程标准：

- 页面根容器宽度、背景色、模块边界。
- 从窗口根到 Item 根的背景归属关系。
- 短内容、空列表、加载态、隐藏模块和底部空白区域的最终可见颜色。
- 每个可见模块的外边距、内边距、宽高、圆角和背景。
- 每组相邻模块的上下间距，以及间距由哪一层代码提供。
- 标题左侧图标、右侧操作、箭头、分割线、角标、标签等结构元素是否真实存在于蓝湖图层。
- 文字的字体族、字号、行高、字重、颜色和截断规则。
- 每个文本和图标颜色所属的语义角色，例如标题、正文、元信息、禁用态、右侧操作和操作箭头。
- 运行态图片、服务端图标和占位图是否按设计尺寸、裁剪方式和圆角变换加载。
- Design Token 名称，而不只是渲染后的十六进制硬编码。

## 核心方法：引导 AI 建立页面级盒模型

UI 缺陷反复修改却难以修复的根本原因，通常是只关注了子控件属性，而忽略了页面级结构的约束。在向 AI 委派审核任务时，应当首先要求其建立一张页面级盒模型检查表，将设计预设值、当前代码实现与项目基准列出对比。

| 属性 | 蓝湖规范 | 静态代码提取值 | 审核判定 | AI 处理建议 |
|---|---|---|---|---|
| 底部空白区域底色 | 页面背景 Token | 父容器使用旧背景资源，列表透明 | 空列表时露出错误底色 | 将页面背景放到父容器 |
| 顶部 Tab 到首个模块间距 | `12dp` | Tab padding 与首个模块 margin 叠加 | 间距来源不清晰 | 建议保留单一间距归属 |
| 首卡到次卡间距 | `8dp` | Provider 插入 `8dp` Spacer | 符合模块间距基准 | 保持 Provider 间距 |
| 页面卡片左右边距 | `8dp` | `16dp` | 不符合同页基准 `8dp` | 建议修改为 `8dp` |
| 卡片背景 | 卡片背景 Token | 旧卡片背景资源 | Token 语义不匹配 | 建议替换为语义匹配资源 |
| 标题左侧图标 | 蓝湖无图标图层 | 旧版 `ImageView` 仍展示 | 多余结构元素 | 建议删除图标和关联 margin |
| 标题字重 | `Medium` | `TextView` 默认 | 缺失字重声明 | 建议添加 `font_weight="medium"` |
| 右侧操作文字与箭头 | 次级文本 Token | 文字和箭头都使用主文本色 | 操作区颜色角色错误 | 只修改右侧操作文字和箭头 |
| 运行态缩略图圆角 | `6dp` | XML 有尺寸，加载器无圆角变换 | 图片本体未还原圆角 | 建议使用项目圆角加载器 |
| 卡片角标文本 | `10sp`、居中 | 默认 Font Padding | 小尺寸文本偏上 | 建议设置行高、字重和 `includeFontPadding=false` |

通过明确指令约束 AI 生成该检查表，不仅能覆盖内部的文本和图片节点，更能强制 AI 检查页面背景、Header 高度、列表视口边距等全局维度。完成盒模型的结构化梳理后，AI 就能准确界定 UI 偏差究竟属于间距、排版、约束关系，还是资源类型的错误。

## 模块间距：先写台账，再改内部细节

最新的蓝湖验收流程将模块间距提升为一等检查对象。只要目标页面存在两个以上模块、卡片或区块，就不能只检查单个模块内部的图标、文字和圆角。相邻模块之间的可见距离，必须先从蓝湖结构化坐标中算出，再回到代码中定位真实来源。

原因很直接：同一个 `8dp` 间距，可能来自上一个模块的 bottom padding、下一个模块的 top margin、列表 Provider 的 Spacer、ItemDecoration、父容器 padding，或者透明父容器露出的背景区域。若不先确认归属，修复内部元素很容易造成二次偏差。

模块间距台账至少应包含以下字段：

| 模块关系 | 蓝湖间距 | 代码来源 | 当前间距 | 归属判断 |
|---|---:|---|---:|---|
| 顶部 Tab -> 功能区 | `12dp` | 页面容器 padding + 首个 Provider | `?dp` | 需要拆分计算 |
| 功能区 -> 首个卡片 | `-1dp` | 首卡 Provider `marginTopDp(8f)` | `8dp` | 修改首卡顶距 |
| 首个卡片 -> 第二个卡片 | `8dp` | 第二个 Provider `marginTopDp(8f)` | `8dp` | 保持 Provider 间距 |

其中 `?dp` 不能留到结论阶段。对于列表或网格页面，还要把 Header 底部残留空间、Provider Spacer、RecyclerView/ListView padding、Item 根 margin、首个可见子控件 margin 一起求和。只有写出类似 `8 + 10 + 12 = 30dp` 的计算过程，才能证明间距判断可复查。

这一步应该早于内部元素修复。页面整体、边距、间距类问题若缺少模块间距台账，就只能算作一次不完整审核。

## 背景归属：底色不是列表项属性

新版蓝湖验收流程中，背景和底色需要作为一等检查对象处理。页面底色并不总是由当前可见的卡片或列表项决定，真正需要审查的是从外到内的背景归属链。

典型背景归属链如下：

```text
Activity/window root
-> Fragment/page root
-> Tab/page container
-> refresh/layout parent
-> RecyclerView/ListView/ScrollView
-> item/module root
```

每一层都需要记录三件事：是否显式设置背景、透明时继承哪一层颜色、最终暴露给用户的颜色是什么。尤其在 Android 列表页中，RecyclerView 通常应保持透明，页面底色由父容器或页面根节点承担；卡片和模块背景则属于 Item 或模块根节点。把页面底色直接涂到列表控件上，可能掩盖短内容和空状态问题，也可能破坏同一页面内不同 Tab 的背景语义。

底部空白区域是最容易漏掉的检查点。短内容、空列表、加载态、模块隐藏、最后一个 Item 下方 padding、列表 overscroll，都可能暴露真实页面背景。只验证长列表满屏状态，不能证明页面底色正确。

因此，背景检查至少应覆盖：

- Activity 根、Fragment 根、Tab 容器、刷新容器、列表控件和 Item 根节点。
- 短内容、空列表、加载态、隐藏模块和 Tab 切换后的可见底色。
- RecyclerView/ListView/ScrollView 的透明默认值，以及父容器是否承担设计稿要求的页面背景。
- 每个背景 Token 的日间、夜间或皮肤覆盖资源。

## Design Token：将语义审核交给代码分析

当蓝湖数据包含命名 Token 时，必须要求 AI 将 Token 名称作为还原规格。

例如，蓝湖给出一个「卡片背景 Token」，代码中同时存在「语义匹配资源」和「旧版近似资源」。二者当前渲染值可能相同，但语义并不一定相同。

如果项目色彩规范未明确声明两者互为别名，AI 在执行静态代码分析时，应找出使用了非语义化资源的节点，并给出警告。在主题切换、深色模式适配等场景下，这种基于语义的自动化审查，能提前拦截因硬编码导致的分化问题。

背景 Token 也要按同样原则处理。即使两个资源在当前皮肤下看起来接近，只要设计稿指定的是页面背景 Token，就不能用相近旧资源或截图采样色替代。Token 的意义在于跨主题保持语义稳定，而不是只保证当前截图像素相近。

## 颜色角色：同一行不等于同一语义

最新流程对颜色审核增加了一个关键约束：颜色不能只按十六进制值、父容器或同一行布局来判断，必须按语义角色拆开。

一个模块 Header 中，标题文字、右侧操作文字、操作箭头和分割线可能在同一个布局文件、同一行容器甚至共享旧资源名，但它们不是同一个角色。修复右侧操作文字颜色时，不能顺手修改标题颜色；修复箭头 tint 时，也不能把正文、元信息或禁用态文字一起替换。

因此，颜色审核表需要额外记录两列：

| 视图身份 | 语义角色 | 蓝湖 Token 或颜色 | 当前代码 | 修复边界 |
|---|---|---|---|---|
| `tv_title` | 模块标题 | 主文本 Token | 主文本资源 | 保持 |
| `tv_more` | 右侧操作文字 | 次级文本 Token | 主文本资源 | 只改操作文字 |
| `drawableRight` | 右侧操作箭头 | 次级图标 Token | 主文本资源 | 与操作文字一起改 |

同类组件基准也要按角色比较。一个 Peer 模块的标题色只能证明标题色惯例，不能证明右侧操作色；右侧操作文字和箭头应当找同角色基准。若 Peer 之间存在差异，优先回到蓝湖 Token 与项目资源映射，并在结论中写明保留差异的原因。

如果蓝湖只返回原始颜色值，也不能直接硬编码或找一个肉眼相近资源。应先查项目资源、生成后的 merged resources 或色彩规范，确认这个值对应的语义资源，再决定代码修改。

## 运行态验证：跨越静态代码的边界

Android 的 XML、Flutter 的 Widget 树或 HarmonyOS 的 ArkUI 声明只能反映静态结构。界面的最终呈现还会受到数据绑定、运行时样式覆盖以及视图隐藏状态的改变。

利用 AI 辅助审核时，可以将验收范围延展至运行态：

- **数据绑定追踪**：让 AI 顺着 ViewModel 或 Controller 的执行路径，检查运行时是否会动态追加前缀、后缀或空状态占位文案。
- **动态修饰符**：检查字重属性是否在代码中被 `setTypeface` 或 ArkUI 的修饰符错误覆盖。
- **列表间距溯源**：在 RecyclerView 或 Flutter ListView 中，项间距可能由 Item 根布局、ItemDecoration 或分割线机制共同作用。AI 能够跨文件检索相关的 Provider 模型与装饰器代码，避免人工审核极易漏掉的间距生成源。
- **空白态背景追踪**：列表内容不足一屏时，最终可见背景来自父容器、刷新布局还是列表控件本身，需要沿布局绑定、适配器和容器层级追到运行态。
- **运行态图片追踪**：服务端下发图标或缩略图时，要顺着布局 `ImageView`、Adapter 绑定、图片加载器和占位图查到最终调用。父容器有圆角并不代表图片本体有圆角，除非父容器真实裁剪子元素，或加载器应用了圆角变换。

## 结构元素：先查存在，再查样式

视觉走查常把标题左图标、右侧箭头、清除按钮、角标、标签和分割线当成「装饰」。在自动化验收中，这些元素需要单独检查。第一步不是判断大小和颜色，而是确认它们是否存在于蓝湖图层。

典型漏检包括：

- 设计稿中的标题行已经只有文字，代码仍保留旧版标题左侧热度图标。
- 右侧操作文案、箭头或清除按钮在某个状态下多出或缺失。
- 卡片内部的排名角标、`NEW` 标识、状态标签、股票标签被误认为顶部 Tab 或筛选标签。
- 服务端图标只核对了 XML 宽高，没有核对加载器是否处理圆角、裁剪和占位图。

因此，AI 审核需要把标题行结构和卡片内部结构拆开。标题左图标、右侧操作、箭头、分割线属于模块 Header 层；排名角标、角落丝带、状态胶囊、覆盖标签属于 Item 或卡片层。若用户只说「标签」，审核流程也应先定位具体图层或代码控件，再决定它是 Tab、筛选项、角标还是卡片内状态标签。

## 补丁约束：修改后先读 diff

AI 自动修复视觉问题时，风险不只在于改错值，还在于改得过宽。颜色资源尤其容易出现这种问题：同一个旧资源名可能同时用于标题、正文、右侧操作和箭头。一次全局替换可以通过编译，却把原本正确的相邻角色一起改掉。

因此，补丁前需要先写出目标清单：

- 文件路径。
- 视图 ID 或绑定名。
- 属性名。
- 旧资源或旧值。
- 新资源或新值。
- 语义角色。

补丁后必须先读 diff，再运行构建。diff 至少要证明三件事：只有目标语义角色发生变化；同一行中的标题、正文、元信息等相邻角色没有被误改；需要成对变化的元素已经一起处理，例如右侧操作文字与右侧操作箭头。

如果 diff 显示修复右侧操作颜色时顺带改了模块标题，审核还没有完成。此时应先回退误改部分，再继续验证。

## 平台映射：为 AI 设定本地规则基准

将蓝湖的抽象设计数据交由 AI 审核前，必须将各平台的单位规范和控件原生能力作为系统级 Prompt（System Prompt）注入。

| 平台 | 几何与间距 | 字体 | AI 重点检索目标 |
|---|---|---|---|
| Android View/XML | `dp` | `sp` | `layout_margin`、约束关系、运行时 Typeface、列表父容器背景 |
| Flutter | logical pixels | `TextStyle` | `EdgeInsets`、`SizedBox`、`FontWeight` |
| HarmonyOS | `vp` | `fp` | `.margin`、`.padding`、父布局约束 |

字重（Font Weight）还原是高频误差项。若基础控件仅支持 `normal` 和 `bold` 的粗略枚举，AI 可以自动参考同类页面的工程实现惯例，提供符合项目历史规范的替换建议，而非生硬报错。

对 Android 列表页还要补充四条规则：

- 不要只看 Item XML。需要检查生成或绑定出来的根布局、刷新容器、RecyclerView/ListView 背景、`clipToPadding`、padding，以及 Adapter 内容少于视口高度时露出的区域。若设计稿要求的是页面背景，通常应由父容器或页面根节点承担，而不是由透明列表控件或单个 Item 承担。
- Header 到首个重复 Item 的距离必须按最终可见距离计算，不能只看某一个 XML margin。Header 残留空间、Provider Spacer、RecyclerView/ListView padding、Item 根 margin 和首个子控件 margin 都要计入。
- 小尺寸角标或排名标签需要单独核对 `includeFontPadding`、line height、gravity、字重和固定高度。容器高度低于 `16dp` 时，默认 Font Metrics 很容易导致文本视觉偏上。
- 文本颜色和图标 tint 需要按视图身份与语义角色分开检查。模块标题、正文、元信息、右侧操作文字、右侧操作箭头不能因为处在同一行就使用同一个判断。

## 标准化 AI 验收执行清单

在日常工程实践中，建议按以下标准化顺序构建自动化的 UI 验收脚本或 Agent 工作流：

1. **上下文注入**：向 AI 明确目标页面、交互状态、应用主题及对应的蓝湖设计节点 ID。
2. **结构化提取**：利用 API 或自动化工具获取精确节点的 `HTML/CSS/Token` 规格。
3. **建立校验表**：指令 AI 构建页面级盒模型检查表，全面覆盖容器约束、背景归属、排版格式及 Design Token。
4. **模块间距台账**：按页面从上到下列出所有相邻模块，记录蓝湖间距、代码来源、当前间距和归属判断。
5. **背景归属审计**：沿窗口根、页面根、Tab 容器、刷新容器、列表和 Item 根节点记录背景来源，覆盖短内容和空列表状态。
6. **颜色角色审计**：按视图身份和语义角色记录标题、正文、元信息、禁用态、右侧操作文字和箭头的 Token 映射。
7. **结构元素检查**：单独检查标题左图标、右侧操作、箭头、角标、标签、分割线、清除按钮和运行态图片加载方式。
8. **基准核对**：AI 自动抽查项目内 1 至 3 个同类组件，确认本地代码的实现惯例。颜色问题只比较同一语义角色。
9. **精准定位**：AI 输出视觉偏差分类，如间距异常、背景归属错误、颜色角色错误、约束关系错误、资源调用不当或结构元素多缺。
10. **自动化修复**：让 AI 严格按单一视觉维度，输出小范围、细粒度的代码修正补丁。
11. **Diff 复核**：构建前先检查补丁 diff，确认没有误改同一行或同一布局内的相邻语义角色。
12. **回归验证**：运行本地 UI 静态检查脚本或抓取设备截图提供最终结果比对。

结论输出时，AI 报告至少需明确交代以下工程事实：

- 页面级全局边距和各主要模块根节点的属性已全部核对完毕。
- 相邻模块的间距台账已经完成，每组可见模块都有明确代码归属。
- 列表组件涉及的动态机制（Provider/ItemDecoration 等）已通过跨文件逻辑检查。
- 页面背景归属链、短内容底部空白区域和空列表状态已完成检查。
- 颜色修改已按标题、正文、元信息、右侧操作文字和箭头等角色分别核对。
- 标题行结构元素、卡片内角标/标签、可见文本和图标资源的加载方式，已完成从布局到运行逻辑的完整追溯。
- 补丁 diff 已完成复核，没有误改相邻语义角色。

## 总结

让 AI 参与 UI 验收，本质上是将感官层面的「看图找不同」，升级为逻辑严密、可追踪的工程化代码审核。

构建可靠 AI 审核流程的核心在于：

- 坚持以结构化数据作为唯一规格来源，严禁使用视觉截图来覆盖或推测 Token、Layout 和 Layer 的真实属性。
- 遵循自顶向下的检查顺序，优先审查页面级盒模型、模块间距台账与容器约束，其次再排查子控件细节。
- 把背景和底色纳入页面级审计，沿窗口根、页面根、Tab 容器、刷新容器、列表和 Item 根节点确认归属。
- 把颜色拆成可追踪的语义角色，避免用同一行、同一父容器或相同旧资源名替代真实设计含义。
- 把结构元素存在性纳入审计，分别检查标题行装饰、右侧操作、卡片角标、状态标签、运行态图片和分割线。
- 突破静态代码文件的限制，将布局追踪至运行态的数据绑定、列表动态生成逻辑以及主题资源的加载过程。
- 在构建和截图前先复核 diff，确认自动补丁没有扩大到无关控件。

通过上述工程化流程的处理，UI 缺陷的拦截将真正摆脱人工直觉的不可靠性。每一次布局改动都可以在代码评审（Code Review）阶段被 AI 代理自动拦截，并提供具备设计理论值和代码坐标的修正建议。唯有如此，高精度的视觉还原才能在复杂的业务迭代中保持极高的稳定性。
